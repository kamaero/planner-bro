# PlannerBro — Пентест-режим отчёт

Дата аудита: 2026-03-11
Среда: production VPS (`plannerbro.ru`)
Метод: конфигурационный hardening-аудит + проверка эксплуатационных журналов

## Executive Summary

- Критичных уязвимостей класса “красные дыры” на уровне базовой инфраструктуры не выявлено.
- Основные векторы (парольный SSH, лишние внешние порты, отсутствие brute-force защиты) закрыты.
- Остаточный риск: умеренный (типичный для публичного VPS с открытым SSH 22).

Итоговый статус: **PASS (с не-критичными рекомендациями)**

## Область проверки

- SSH/доступ к серверу
- UFW/firewall
- fail2ban/anti-bruteforce
- открытые порты и Docker-публикации
- TLS/сертификаты
- базовые sysctl hardening параметры
- автообновления безопасности
- эксплуатационные auth-логи

## Наблюдения по журналам

- На SSH наблюдается постоянный шум brute-force по `root/ubuntu/invalid user`.
- Успешные логины происходят только по public key.
- Признаков успешного password-login не выявлено.

## Оставшиеся риски (не критичные)

1. SSH (22/tcp) остаётся публичным endpoint.
- Риск: постоянные брутфорс-попытки (снижается fail2ban и key-only auth).

2. Root SSH доступ по ключу разрешён.
- Риск: при компрометации ключа root сразу даёт полный доступ.

3. `plannerops` в группе sudo.
- Это осознанный резервный канал доступа; требуется бережное управление ключами и паролем.

## Рекомендации (без “перезажима”)

### Рекомендуемый минимум (операционный)

- Контролировать список authorized_keys раз в месяц.
- Ротировать SSH-ключи для админов раз в 90 дней.
- Проверять auth-log и fail2ban weekly.
- Перед пентестом запускать этот чеклист повторно.

### Опционально (если захотите усилить позже)

- Перевести работу на `plannerops` + `PermitRootLogin no`.
- Ограничить SSH по source IP/VPN.
- Подключить MFA для SSH (PAM TOTP).

На дату 2026-03-11 сервер соответствует практическому baseline hardening для публичного production VPS и не содержит очевидных критичных “красных дыр” в проверенной зоне.
