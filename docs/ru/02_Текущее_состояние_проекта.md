# PlannerBro — Текущее состояние проекта

Обновлено: 2026-03-19

## Зачем нужен этот документ

Это оперативный снимок проекта.

Он отвечает на вопросы:

- что уже есть в системе;
- что сейчас главное;
- где есть расхождения;
- на что можно опираться как на текущую реальность, а не на мечту.

## Кратко

PlannerBro уже является рабочей многослойной системой с backend, web, mobile, realtime, уведомлениями, импортом, базовой аналитикой, хранилищем файлов и ролевой моделью.

Главная текущая проблема не в отсутствии фич, а в росте стоимости понимания и изменения проекта:

- бизнес-логика частично распухла в крупных файлах;
- доменные правила местами дублируются;
- документация нуждается в роли внешней памяти, а не только справочника;
- процесс проверки качества нужно сделать более воспроизводимым.

## Git и синхронизация

На момент фиксации состояния:

- локальная ветка уже ушла вперёд относительно `origin` на рабочие архитектурные коммиты;
- рабочая директория локально не чистая;
- значит, состояние “что в ветке” и состояние “что лежит в локальной папке” различаются.

Практический вывод:

- перед деплоем и перед переключением между Mac нужно отдельно проверять не только ветку, но и рабочую директорию.

## Что уже реализовано

Крупные функциональные блоки, которые уже присутствуют в коде:

- управление проектами и задачами;
- роли, visibility scope и отдельные permissions;
- уведомления in-app, email, Telegram, mobile push;
- sidebar с presence и системной активностью;
- импорт исходных файлов и AI ingestion;
- Flutter mobile-контур;
- encrypted vault;
- healthcheck, backup scripts, deploy scripts;
- встроенная справка и UX-подсказки.

## Главный фокус на сейчас

Текущий правильный фокус проекта:

1. Внешняя память проекта через документацию.
2. Стабилизация доменных контрактов.
3. Упрощение самых тяжёлых модулей.
4. Усиление воспроизводимых проверок.

## Архитектурные наблюдения

### Что уже хорошо

- стек понятный и современный;
- backend, web и mobile уже интегрированы в единую систему;
- есть зачатки сервисного слоя и явной ролевой модели;
- есть документация по продукту и эксплуатации;
- фронтенд-проверки `typecheck` и `lint` проходят.

### Что уже начинает скрипеть

- крупные backend-роуты и frontend-страницы перегружены;
- часть бизнес-логики живёт не там, где её ожидаешь;
- контракты статусов, ролей и прав размазаны по нескольким слоям;
- backend-проверки уже получили облегчённый smoke path, но пока ещё не стали полностью “нулевыми” по подготовке;
- realtime/presence пока рассчитаны на single-process сценарий.

### Что уже начали исправлять

Первые шаги по разгрузке тяжёлых модулей уже начаты:

- для `backend/app/api/v1/projects.py` вынесена первая группа helper-правил в отдельный сервис `backend/app/services/project_rules_service.py`;
- для `backend/app/api/v1/projects.py` вынесен отдельный access/helper слой в `backend/app/services/project_access_service.py`;
- для AI draft approval внутри projects-flow вынесен отдельный use-case сервис `backend/app/services/project_ai_draft_service.py`;
- для `backend/app/api/v1/tasks.py` вынесены task planning/dependency rules в `backend/app/services/task_rules_service.py`;
- для frontend вынесены общие словари статусов и приоритетов в `frontend/src/lib/domainMeta.ts`;
- files/AI блок страницы проекта вынесен из `frontend/src/pages/ProjectDetail.tsx` в `frontend/src/components/ProjectFilesSection/ProjectFilesSection.tsx`;
- toolbar списка задач вынесен из `frontend/src/pages/ProjectDetail.tsx` в `frontend/src/components/ProjectTaskListToolbar/ProjectTaskListToolbar.tsx`;
- для этой вынесенной логики добавлен отдельный smoke-тест.

## Самые тяжёлые узлы

Наиболее перегруженные по размеру и ответственности файлы:

- `backend/app/api/v1/projects.py`
- `backend/app/api/v1/tasks.py`
- `frontend/src/pages/ProjectDetail.tsx`
- `frontend/src/pages/Team.tsx`
- `frontend/src/pages/Dashboard.tsx`
- `frontend/src/App.tsx`

Размер сам по себе не преступление. Но когда файл растёт и одновременно отвечает за много разных вещей, он становится дорогим в сопровождении.

## Известные риски

### 1. Drift между локальным состоянием, git и продом

Если рабочая директория не чистая, можно получить разные результаты для:

- git-based deploy;
- deploy из локальной папки;
- ручной диагностики проблемы.

### 2. Дублирование доменных правил

Статусы, приоритеты, права и label-словари местами повторяются в:

- backend;
- frontend;
- mobile.

Это повышает шанс тихого рассинхрона.

### 3. Ограничение realtime

Presence и часть realtime-координации зависят от in-memory состояния одного процесса backend.

Пока сервер один, это допустимо.
При горизонтальном масштабировании это станет архитектурным ограничением.

### 4. Проверки качества

Фронтенд-проверки уже предсказуемы.
Backend-проверки нужно довести до официального и воспроизводимого сценария.
