# PlannerBro — Проверки и верификация

Обновлено: 2026-03-19

## Зачем нужен этот документ

Этот файл отвечает на вопрос:

как проверить проект так, чтобы результат был воспроизводимым, а не “у меня вроде открылось”.

Верификация здесь означает не магию, а обычную инженерную проверку:

- код собирается;
- ключевые проверки проходят;
- после деплоя система отвечает;
- мы понимаем, что именно проверили, а что нет.

## Принцип проверки

Проверки делятся на три уровня:

1. Быстрые локальные проверки.
2. Полные локальные проверки.
3. Smoke-проверка после деплоя.

## 1. Быстрые локальные проверки

Это минимальный набор перед коммитом или после небольшого изменения.

### Единый вход

Из корня проекта:

```bash
./scripts/check-local.sh
```

Что делает этот скрипт сейчас:

- запускает frontend `typecheck`;
- запускает frontend `lint`;
- проверяет наличие backend Python-зависимостей;
- запускает backend smoke suite через `python -m unittest discover -s tests -p '*_smoke.py' -q`.

Если backend-зависимости не установлены, скрипт завершится с понятной ошибкой и подскажет следующую команду.

### Подготовка backend-окружения для проверок

Из корня проекта:

```bash
./scripts/setup-backend-check-env.sh
```

Что делает этот скрипт:

- создаёт `.venv-check312`;
- использует `python3.12`;
- ставит облегчённые backend-зависимости из `backend/requirements-check.txt`.

Зачем нужен отдельный `requirements-check.txt`:

- smoke-проверкам не нужен весь production-набор библиотек;
- это ускоряет подготовку окружения;
- базовая локальная проверка перестаёт тащить Celery, Firebase и прочий “оркестр с литаврами”, если они не участвуют в текущем smoke-наборе.

Почему именно `python3.12`:

- backend-стек проекта рассчитан на Python 3.12;
- системный Python 3.14 на этой машине не подходит для бесшовной установки части зависимостей проверки.

### Frontend вручную

Если нужно прогнать только frontend, запускать из `frontend/`:

```bash
npm run typecheck
npm run lint
```

Что это проверяет:

- TypeScript-совместимость;
- базовые статические ошибки;
- lint-правила.

### Backend вручную

На текущем состоянии проекта backend-проверки уже имеют единый вход и отдельное лёгкое check-окружение.

Что уже есть сейчас:

- smoke-тесты в `backend/tests/`;
- единый локальный вход через `./scripts/check-local.sh`.
- скрипт подготовки backend-окружения `./scripts/setup-backend-check-env.sh`.
- отдельный набор зависимостей `backend/requirements-check.txt`.

Что важно понимать:

- для их запуска всё ещё нужны backend-зависимости;
- но теперь это отдельный облегчённый набор, а не полный production-стек;
- `check-local.sh` запускает smoke-тесты с test-env переменными `DEBUG=true` и `SECRET_KEY=local-check-secret`, чтобы локальная проверка не падала только из-за отсутствующего `.env`.

## 2. Полные локальные проверки

Это набор перед серьёзным изменением, merge или подготовкой к deploy.

### Docker-based запуск системы

Из корня проекта:

```bash
cp .env.example .env
docker-compose up --build
```

Что после этого проверить:

- web UI доступен;
- `/health` отвечает;
- backend поднялся с миграциями;
- frontend собрался;
- базовые пользовательские сценарии работают.

### Backend tests

Текущий честный статус:

- тесты существуют;
- backend smoke-проверки уже имеют официальный локальный вход;
- это отдельная задача в рабочем TODO.

До нормализации процесса проверок нужно явно фиксировать:

- в каком окружении запускали;
- какие зависимости были установлены;
- что реально удалось прогнать.

## 3. Smoke-проверка после deploy

Для продовой быстрой проверки есть готовый скрипт:

```bash
./scripts/smoke-check-prod.sh
```

По умолчанию он смотрит в:

- `BASE_URL=https://plannerbro.ru`

Что он проверяет:

- главная страница отвечает;
- `team-board` отвечает;
- `/health` отвечает;
- health payload сообщает, что `db` и `redis` в порядке;
- frontend shell содержит root node;
- основной bundle загружается и содержит ожидаемые маркеры;
- при наличии логина и пароля можно проверить аутентифицированный сценарий.

### Полная smoke-проверка с логином

Можно передать переменные:

```bash
LOGIN_EMAIL=... LOGIN_PASSWORD=... ./scripts/smoke-check-prod.sh
```

Или через `SMOKE_EMAIL` / `SMOKE_PASSWORD`.

## Что сейчас уже можно считать проверенным

На момент создания этого документа:

- frontend `typecheck` проходит;
- frontend `lint` проходит;
- продовый smoke-скрипт в репозитории есть и формализован;
- backend smoke-проверки имеют воспроизводимый запуск через `./scripts/check-local.sh`, но полное разделение на CI / pre-deploy / post-deploy всё ещё нужно развивать.

## Что пока ещё не доведено до идеала

- полностью бесшовный backend check без предварительного `setup-backend-check-env.sh`;
- явное разделение на dev / CI / pre-deploy / post-deploy verification;
- контрактные тесты для ролей, статусов, зависимостей и planning modes.

## Минимальный практический чеклист перед merge

- [ ] Обновлён нужный код.
- [ ] Обновлена документация, если менялся процесс, архитектура или текущее состояние.
- [ ] Для frontend выполнены `npm run typecheck` и `npm run lint`.
- [ ] Для backend честно зафиксировано, что именно проверено и в каком окружении.
- [ ] Понятно, какой следующий риск остаётся непокрытым.

## Минимальный практический чеклист после deploy

- [ ] Запущен `./scripts/smoke-check-prod.sh`.
- [ ] Проверен `/health`.
- [ ] Проверен web shell.
- [ ] При возможности проверен логин и `/api/v1/users/me`.
- [ ] Зафиксирован результат проверки без фантазии и украшений.
