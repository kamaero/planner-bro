# PlannerBro — Доменные контракты

Обновлено: 2026-03-19

Этот документ фиксирует "канонические словари" и контракты событий, чтобы backend/web/mobile говорили на одном языке.

## 1. Единый словарь статусов проектов

Канон (backend enum `project_status`):

- `planning`
- `tz`
- `active`
- `testing`
- `on_hold`
- `completed`

UI-метки (web):

- `planning` -> `Планирование`
- `tz` -> `ТЗ`
- `active` -> `Активный`
- `testing` -> `Тестирование`
- `on_hold` -> `Пауза`
- `completed` -> `Завершён`

Где закреплено:

- backend: `backend/app/models/project.py`
- web labels/options: `frontend/src/lib/domainMeta.ts`
- web type: `frontend/src/types/index.ts`

## 2. Единый словарь статусов задач

Канон (backend enum `task_status`):

- `planning`
- `tz`
- `todo`
- `in_progress`
- `testing`
- `review`
- `done`

UI-метки (web):

- `planning` -> `Планирование`
- `tz` -> `ТЗ`
- `todo` -> `К выполнению`
- `in_progress` -> `В работе`
- `testing` -> `Тестирование`
- `review` -> `На проверке`

Канонический порядок для доски/таблиц (web):

- `planning -> tz -> todo -> in_progress -> testing -> review -> done`

Где закреплено:

- backend: `backend/app/models/task.py`
- web labels/order: `frontend/src/lib/domainMeta.ts`
- web type: `frontend/src/types/index.ts`

## 3. Единый словарь приоритетов

Канон (backend enum `project_priority` и `task_priority`):

- `low`
- `medium`
- `high`
- `critical`

UI-метки (web):

- `low` -> `Низкий`
- `medium` -> `Средний`
- `high` -> `Высокий`
- `critical` -> `Критический`

Спец-правило домена:

- если `control_ski=true`, то приоритет трактуется как `critical`.

Где закреплено:

- backend enum: `backend/app/models/project.py`, `backend/app/models/task.py`
- backend rule: `backend/app/services/project_rules_service.py`, `backend/app/services/task_rules_service.py`
- web labels/order/colors: `frontend/src/lib/domainMeta.ts`

## 4. Матрица ролей и visibility scope

Роли (`user_role`):

- `admin`
- `manager`
- `developer`

Visibility scope (`user_visibility_scope`):

- `full_scope`
- `department_scope`
- `own_tasks_only`

Канонические правила доступа:

- `admin` или `full_scope` -> глобальный доступ.
- `own_tasks_only` -> видимость только своих задач/контекста.
- `department_scope` -> доступ по дереву отделов + подчиненности.

Упрощенная матрица:

| role/scope | Что видит по умолчанию |
|---|---|
| `admin` (+ любой scope) | Все проекты/задачи/пользователей |
| `manager` + `full_scope` | Практически глобальный контур (кроме admin-only операций) |
| `manager` + `department_scope` | Свой контур отдела/подчиненных |
| `developer` + `department_scope` | Свой контур отдела/подчиненных |
| `developer` + `own_tasks_only` | В основном свои задачи |

Где закреплено:

- backend enum и поля: `backend/app/models/user.py`
- алгоритм scope: `backend/app/services/access_scope.py`
- подробные сценарии: `docs/ru/15_Права_роли_и_назначения.md`

## 5. Контракт отдельных permissions

Поля-флаги в модели пользователя:

- `can_delete`
- `can_import`
- `can_bulk_edit`
- `can_manage_team`

Смысл флагов:

- `can_delete` -> разрешает удаление сущностей там, где проверка завязана на этот флаг.
- `can_import` -> разрешает импорт задач/файловых сценариев.
- `can_bulk_edit` -> разрешает bulk-операции над задачами.
- `can_manage_team` -> расширяет управление пользователями/назначениями.

Важное правило:

- UI может скрывать кнопки, но финальное решение всегда за backend-проверкой.

Где закреплено:

- модель/схемы: `backend/app/models/user.py`, `backend/app/schemas/user.py`
- backend проверки: `backend/app/api/v1/projects.py`, `backend/app/api/v1/tasks.py`, `backend/app/services/access_scope.py`

## 6. Контракт realtime / notification events

Канонический набор websocket events:

- `task_created`
- `task_updated`
- `task_assigned`
- `project_updated`
- `deadline_warning`
- `escalation_sla_breached`
- `ai_drafts_ready`
- `ai_drafts_failed`
- `team_status_reminder`
- `chat_message`

Общий wire-формат:

- `{ "event": "<event_name>", "data": { ... } }`

Минимальные payload-поля:

- `task_created` -> `task_id`, `project_id`
- `task_updated` -> `task_id`, `project_id` (в части сценариев может приходить расширенный `data.kind`)
- `task_assigned` -> `notification_id`, `task_id`, `title`, `body`
- `project_updated` -> `project_id` (в user-targeted сценариях также `notification_id`, `title`, `body`)
- `deadline_warning` -> `task_id`, `project_id`, `deadline_key`
- `escalation_sla_breached` -> `task_id`, `project_id`, `breached_at`
- `ai_drafts_ready` -> `project_id`, `project_file_id`, `job_id`, `drafts_count`
- `ai_drafts_failed` -> `project_id`, `project_file_id`, `job_id`, `error`
- `team_status_reminder` -> `notification_id`, `title`, `body`
- `chat_message` -> сериализованный `ChatMessageOut`

Где закреплено:

- backend event names: `backend/app/services/events.py`
- backend producers: `backend/app/services/notification_service.py`, `backend/app/tasks/ai_ingestion.py`, `backend/app/api/v1/chat.py`
- web event constants: `frontend/src/api/events.ts`
- web consumer logic: `frontend/src/hooks/useWebSocket.ts`

## 7. Правило изменения контракта

При любом изменении enum/permission/event нужно одновременно обновить:

- backend модель/схему и producer-код;
- web constants/types/handlers;
- этот документ (`20_Доменные_контракты.md`);
- TODO/план, если изменился приоритет или объем техдолга.
