# PlannerBro — Workflow документации и синхронизации

Обновлено: 2026-03-19

## Зачем нужен этот документ

Этот файл описывает, как обновлять документацию и как не допускать расхождения между:

- кодом;
- рабочим планом;
- архитектурной памятью;
- локальной копией;
- репозиторием.

Иначе говоря, это инструкция “как не потерять нить проекта”. Иногда нить тонкая, а ноутбуков больше одного.

## Базовое правило

Если изменился код, который влияет на понимание проекта, нужно обновить документацию в том же изменении.

## Когда документацию обязательно обновлять

Обновление обязательно, если меняется хотя бы одно из этого:

- архитектура модуля;
- структура директорий или точек входа;
- правила ролей, прав или visibility;
- статусы, приоритеты, контракты событий;
- деплой, запуск, smoke-check;
- текущий рабочий приоритет;
- известные риски или ограничение системы.

## Минимальный чеклист перед завершением задачи

- [ ] Код изменён и проверен.
- [ ] Рабочий TODO обновлён.
- [ ] Если менялась архитектура или процесс, документация обновлена.
- [ ] Если изменилось фактическое состояние проекта, обновлён `Current State`.
- [ ] Зафиксирован следующий разумный шаг.

## Правило “не делать уверенную фантастику”

Документация не должна писать “всё проверено”, если это не проверено.

Если что-то не удалось верифицировать, пишем честно:

- что проверили;
- что не проверили;
- почему;
- какой остаётся риск.

Это скучнее, чем героические легенды, зато надёжнее.

## Правило синхронизации с git

Перед началом работы:

1. Проверить текущую ветку.
2. Проверить, чиста ли рабочая директория.
3. Проверить, совпадает ли `HEAD` с нужной удалённой веткой.
4. Только после этого считать локальное состояние понятным.

## Что считать тремя разными состояниями

Важно не смешивать эти сущности:

- `состояние ветки` — какие коммиты в `HEAD`;
- `локальное состояние рабочей директории` — какие есть незакоммиченные изменения;
- `состояние продового сервера` — что реально выкатывается и работает.

Если не разделять эти три слоя, диагностика проблем превращается в детектив с плохим кофе.

## Правило для LLM-coder

Перед большой задачей агент должен:

1. Прочитать портал документации.
2. Прочитать текущий план реализации.
3. Прочитать текущее состояние проекта.
4. Только потом идти в код.

Это уменьшает количество бесполезных блужданий по репозиторию.
