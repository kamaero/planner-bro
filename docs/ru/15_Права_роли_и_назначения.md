# PlannerBro — Права, роли и назначения

Документ фиксирует правила доступа и назначения пользователей в проектах и задачах.

## 1. Базовые сущности прав

- `role`: системная роль пользователя (`admin`, `manager`, `developer`).
- `visibility_scope`: область видимости (`full_scope`, `department_scope`, `own_tasks_only`).
- флаги: `can_manage_team`, `can_delete`, `can_import`, `can_bulk_edit`.
- оргструктура: `department_id`, `manager_id`, руководитель отдела (`Department.head_user_id`).
- должность: `position_title` (используется для специальных сценариев ГИП/ЗАМ).

## 2. Видимость данных

- `admin` / `full_scope`: видит все проекты/задачи/пользователей.
- `department_scope`: видит в рамках своего дерева отделов + подчиненности.
- `own_tasks_only`: видит только свои назначенные задачи и связанные проекты.

## 3. Кто может назначать исполнителей на задачи

Назначение проверяется сервером централизованно.

### Может назначать по всей организации

- `admin`.
- пользователь с `visibility_scope=full_scope`.
- пользователь с должностью (по `position_title`) содержащей:
  - `гип`,
  - `главный инженер проектов`,
  - `зам`,
  - `заместитель`.
- пользователь, который:
  - `role=manager`, или
  - `can_manage_team=true`, или
  - является руководителем отдела (`Department.head_user_id`).

Примечание: для не-админов из общего списка исключается назначение на пользователя с `role=admin`.

### Остальные пользователи

- могут назначать только в пределах своего доступного scope.

## 4. Кто может назначать на проект

Правило назначения на проект синхронизировано с задачами:

- добавление участника проекта (`member`/`manager`) проходит ту же проверку scope назначения;
- передача владельца проекта (`owner_id`) также проходит проверку scope назначения.

Дополнительно:

- назначить роль `owner` через endpoint добавления участника нельзя (только transfer ownership);
- роль `manager` в проекте обычно может назначить owner/admin.

## 5. Матрица (упрощенно)

| Субъект | Видимость | Назначение на задачу | Назначение в проект |
|---|---|---|---|
| admin | Вся система | Любого активного | Любого активного |
| ГИП/ЗАМ (по должности) | По настройке scope | Любого не-admin | Любого не-admin |
| Начальник отдела | Отдел + дерево | Любого не-admin | Любого не-admin |
| manager | Обычно department_scope | Любого не-admin | Любого не-admin |
| developer | own/department scope | В рамках своего scope | В рамках своего scope |

## 6. Практический сценарий

- ГИП создает задачу в проекте отдела А.
- Назначает исполнителя из отдела Б и руководителя из отдела В.
- Система автоматически добавляет назначенных пользователей в `ProjectMember` (роль `member`), если их там не было.

## 7. Что важно для админа

- Для ГИП/ЗАМ корректно заполнить `position_title`.
- Для начальников отделов корректно выставить `Department.head_user_id`.
- Для изоляции исполнителей использовать `visibility_scope=own_tasks_only` и переключатель `own_tasks_visibility_enabled`.
