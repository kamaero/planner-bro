# Чеклист нового продового сервера

Этот документ нужен после переезда на новый прод, чтобы эксплуатация была скучной и предсказуемой.  
Скучная инфраструктура звучит не романтично, зато именно она не будит ночью.

## 1. Зафиксировать целевой сервер

- Боевой хост PlannerBro: `168.222.194.92`.
- Перед любым деплоем проверить, что `plannerbro.ru` и `www.plannerbro.ru` резолвятся именно в этот IP.
- Если нужен временный или тестовый хост, задавать его явно через переменную `REMOTE_HOST`, а не надеяться на память и кофе.

Проверка:

```bash
dig +short plannerbro.ru
dig +short www.plannerbro.ru
```

## 2. Проверить базовый контур сервисов

Нужно убедиться, что на сервере живут именно те контейнеры, которые мы считаем продом.

Проверка:

```bash
ssh root@168.222.194.92
cd /opt/planner-bro
docker compose -f docker-compose.prod.yml ps
```

Ожидаемо должны быть доступны как минимум:

- `frontend`
- `backend`
- `postgres`
- `redis`
- `celery_worker`
- `celery_beat`
- `nginx`
- Excalidraw-сервисы, если whiteboard используется в проде

## 3. Проверить автозапуск после перезагрузки

Идея простая: после `reboot` сервисы должны подняться сами.  
Если что-то требует ручного пинка, это уже не автоматизация, а приручённая магия.

Что проверить:

- `docker` стартует после перезагрузки
- контейнеры имеют корректную политику `restart`
- внешний `nginx` или dockerized `nginx` действительно поднимается сам

## 4. Проверить SSL и внешний доступ

Минимум:

- сайт открывается по `https`
- сертификат выдан для `plannerbro.ru`
- есть понятный способ продлить сертификат

Быстрые проверки:

```bash
curl -I https://plannerbro.ru
curl -I https://plannerbro.ru/team-board
```

## 5. Прогнать smoke-check после деплоя

`smoke-check` это короткая серия проверок "система вообще жива". Не глубокое тестирование, а быстрый осмотр пульса.

После каждого прод-деплоя проверить:

- открывается главная страница
- открывается дашборд без легаси-версии
- отвечает API health endpoint
- работает логин
- открывается `/team-board`
- Excalidraw загружается внутри `iframe` без сообщения `I'm not a pretzel!`

Примеры:

```bash
curl -I https://plannerbro.ru/
curl -I https://plannerbro.ru/team-board
curl -I https://plannerbro.ru/health
```

## 6. Проверить deploy-скрипты

Все штатные скрипты должны по умолчанию указывать на текущий боевой сервер:

- `scripts/deploy-frontend-dist.sh`
- `scripts/deploy-prod-backend.sh`
- `scripts/deploy-excalidraw.sh`
- `scripts/deploy-excalidraw-internal.sh`
- `scripts/backup-postgres.sh`
- `scripts/reset-user-password-vps.sh`

Перед ручным запуском полезно глазами проверить, куда именно летит `REMOTE_HOST`.

## 7. Проверить логи и наблюдаемость

Важно иметь быстрые команды, которые помогают понять "что болит", не устраивая археологическую экспедицию.

Примеры:

```bash
ssh root@168.222.194.92
cd /opt/planner-bro
docker compose -f docker-compose.prod.yml logs -f frontend
docker compose -f docker-compose.prod.yml logs -f backend
docker compose -f docker-compose.prod.yml logs -f nginx
```

Если Excalidraw вынесен в отдельные контейнеры, отдельно смотреть и их логи.

## 8. Проверить резервные копии

Резервная копия базы данных бесполезна, если она только "где-то вроде есть".  
Полезна она тогда, когда мы знаем, где лежит, как создаётся и как восстанавливается.

Проверить:

- работает `scripts/backup-postgres.sh`
- backups складываются в ожидаемую директорию
- ротация старых копий не удаляет всё подряд
- есть понятная инструкция по восстановлению

## 9. Проверить секреты и env-файлы

Нужно понимать, где живут секреты и кто ими управляет.

Минимальный список для ревизии:

- `.env.prod`
- OAuth-секреты
- SMTP-настройки
- Telegram bot token
- `VAULT_ENCRYPTION_KEY`
- пароли Postgres и Redis

## 10. Обновить документацию после изменения инфраструктуры

После любого заметного изменения обновить:

- `README.md`
- `CLAUDE.md`
- `docs/ru/16_Шпаргалка_доступ_к_серверу_с_Mac.md`
- этот чеклист, если появились новые шаги

Главная идея простая: если в документации старый IP, значит в какой-то момент кто-то снова задеплоит "куда-то туда". А нам такой квест больше не нужен.
