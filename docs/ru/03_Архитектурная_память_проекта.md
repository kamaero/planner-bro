# PlannerBro — Архитектурная память проекта

Обновлено: 2026-03-19

## Зачем нужен этот документ

Это короткая карта архитектуры “как есть”.

Не академическая схема ради красоты, а рабочая память для:

- человека, который хочет быстро вспомнить устройство проекта;
- LLM-coder, которому нужно быстро понять границы модулей и не бегать кругами.

## Система целиком

PlannerBro состоит из пяти основных контуров:

1. Backend API и бизнес-логика.
2. Web frontend.
3. Mobile frontend.
4. Background jobs и рассылки.
5. Ops / deploy / инфраструктурный контур.

## Backend

Базовая структура backend:

- `app/core` — конфигурация, безопасность, база, внешние подключения;
- `app/models` — ORM-модели;
- `app/schemas` — API-схемы;
- `app/api/v1` — HTTP-роуты;
- `app/services` — бизнес-логика и вспомогательные сервисы;
- `app/tasks` — фоновые задачи Celery.

### Текущее состояние backend-архитектуры

На уровне намерения архитектура слоистая: роуты должны быть тонкими, а бизнес-логика должна жить в сервисах.

На уровне реальности:

- часть логики уже вынесена в services;
- но значительная доля правил до сих пор живёт прямо в крупных роут-файлах.

Главный вывод:

- архитектурное направление правильное;
- реализация пока не доведена до конца;
- рефакторинг стоит делать эволюционно, а не большим взрывом.

## Frontend

Базовая структура frontend:

- `src/pages` — экранные страницы;
- `src/components` — повторно используемые UI-блоки;
- `src/hooks` — data hooks и поведение;
- `src/api` — клиент API и события;
- `src/store` — auth/theme state;
- `src/lib` — утилиты и форматирование;
- `src/types` — типы и контракты web-слоя.

### Текущее состояние frontend-архитектуры

Сильная сторона:

- hooks и shared components уже существуют;
- роутинг и общий layout понятны.

Слабая сторона:

- некоторые страницы стали слишком тяжёлыми;
- часть доменной логики и словарей живёт прямо внутри экранов;
- крупные страницы сложнее безопасно менять.

## Mobile

Базовая структура mobile:

- `core` — API и инфраструктурные клиенты;
- `models` — модели данных;
- `providers` — состояние и загрузка данных;
- `screens` — экраны;
- `widgets` — повторно используемые элементы.

### Текущее состояние mobile-архитектуры

Mobile-контур уже полезен и практичен:

- авторизация;
- проекты и задачи;
- уведомления;
- аналитика;
- быстрые действия.

Главный риск mobile сейчас не в UI, а в дублировании доменных правил относительно web/backend.

## Realtime и presence

Сейчас realtime держится на WebSocket и локальном in-memory менеджере соединений backend.

Это подходит для текущего single-instance сценария.

Важно помнить:

- это рабочее решение;
- это не бесконечно масштабируемое решение;
- документированное ограничение лучше, чем случайный сюрприз на проде.

## Notification pipeline

Уведомления сейчас распределены по нескольким каналам:

- in-app;
- WebSocket;
- email;
- Telegram;
- mobile push.

Это сильная сторона продукта, но и зона высокой сложности:

- здесь особенно важно держать единый контракт событий;
- здесь особенно опасен скрытый рассинхрон.

## Где проект наиболее хрупок

Хрупкость сейчас выше всего в местах, где сходятся:

- права доступа;
- статусы и переходы;
- назначение исполнителей;
- импорт и AI ingestion;
- уведомления;
- крупные универсальные экраны и роуты.

## Архитектурные принципы на ближайшие итерации

Ниже принципы, которых стоит придерживаться в следующих изменениях:

- один доменный контракт должен иметь один главный источник;
- роуты и страницы должны координировать, а не хранить всю логику внутри себя;
- новый документ важен только если он реально помогает продолжать работу;
- каждое крупное изменение должно уменьшать стоимость понимания системы;
- рефакторинг без проверок — это не рефакторинг, а азартная игра.
